## Источники данных

- **RDF (Repozytorium Dokumentów Finansowych)** — первоисточник финансовой
  отчётности. Спайк сделан: под Angular-фронтом rdf-przegladarka.ms.gov.pl
  есть JSON API, но номер KRS шифруется на клиенте, а прямые запросы банит
  Imperva (бан IP — красная линия, напрямую не скрейпить). Образец отчёта
  CD Projekt лежит в `.playwright-mcp/rdf-spike/`: XBRL/ESEF у биржевых,
  XML Минфина у обычных sp. z o.o.
  **Развилка (решение за владельцем проекта):** Playwright-адаптер /
  остаёмся на агрегаторах (okredo/aleo уже дают цифры) / отложить.
- **GUS REGON** — ключ выдают только по email-заявке, форма рассчитана на
  юрлиц. Даст официальные PKD/размер занятости.
- **Имена правления без звёздочек** — официальный backend wyszukiwarki
  (`wyszukiwarka-krs-api.ms.gov.pl/api/wyszukiwarka/danepodmiotu`) отдаёт
  ПОЛНЫЕ имена zarządu (так их берёт gowork), но хост за Imperva + токен
  `apikey` ротируется → путь только через браузер (Playwright), красная
  линия. НЕ делаем: директор — наёмная фигура, приоритет у владельцев
  (CRBR, уже интегрирован). Оставляем zarząd со звёздочками из dział 2.
- **CRBR — сетевой сигнал по PESEL** (отложено, PII). CRBR отдаёт PESEL
  бенефициаров, но мы его НЕ храним (минимизация PII, аудитория ЕС). При
  наличии защищённого PII-хранилища PESEL позволил бы связывать однодневки
  общим владельцем — сильный сигнал для детекции. Вернуться вместе с
  access-controlled хранилищем (см. «Инфраструктура»).
- **CRBR под нагрузкой** — одиночные и мелкие батчи идут с
  `reCaptchaToken:"0"` без throttling (проверено 16 запросов подряд). Для
  массового прогона (тысячи NIP) добавить задержки/мониторинг recaptcha.
- **Сеть владельцев — профилирование связанных фирм и граф** (отложено,
  явный scope creep). Сеть 1-го уровня уже СДЕЛАНА: обратный поиск CRBR
  по PESEL (`kontekstWyszukania:2`) даёт другие фирмы владельца прямо из
  ответа (название/KRS/форма/адрес/`postepowanie`); в отчёте — список +
  флаг общего адреса. PESEL держится только в памяти, в evidence не
  попадает. ОТЛОЖЕНО дорогое: (1) профилирование КАЖДОЙ связанной фирмы
  по её KRS-одпису (возраст/капитал/отчётность → shell-подобность) —
  +N KRS-запросов на отчёт; (2) рекурсия/граф сети (бенефициары связанных
  фирм, общие директора по всей сети, визуализация). Плюс: reverse по
  имя+дата рождения для иностранцев без PESEL (сейчас у таких сеть не
  проверяется, помечаем «нет PESEL»).
- **Санкции/PEP-скрининг владельцев** (ОПЦИЯ, владелец проекта пока НЕ
  берёт в работу — 2026-07-10). Стало возможным с извлечением личностей
  бенефициаров (имя+гражданство из CRBR): прогон по публичным спискам —
  консолидированный санкционный ЕС, OFAC SDN, UK OFSI, PEP. Решающий
  сигнал для «подписывать ли контракт», особенно при иностранных
  владельцах (на MPSYSTEM — белорусы, а Беларусь под санкциями ЕС).
  Легально (списки публичные). Засада: fuzzy-матч имён (транслитерация
  кириллицы, тёзки) → выдавать «возможное совпадение, проверьте», а не
  ложную уверенность.
- **CRBR — расхождения и история владения** — СДЕЛАНО (2026-07-10).
  Форвардный запрос теперь диапазоном с 13.10.2019: одним вызовом текущий
  состав + история. Флаги: `listaInformacjiORozbieznosciach` (расхождение
  по бенефициару — красный флаг; обработано защитно, живого позитива пока
  не встретили) и недавняя СМЕНА владельцев (гейт на факт смены, чтобы
  свежую фирму не путать с флипом; порог 180 дн.). Проверено: TK FOOD —
  реальная смена (учредитель Gabinecki → белорусы через 11 дней),
  FUNDACJA ACELATUS — свежая без ложного флага. Остаётся на потом: эти
  сигналы как метки на графе владельцев.
- **GoWork — отзывы сотрудников (employee reviews)** — ядро сигнала о
  климате у работодателя; клиентские отзывы Google это НЕ замена.
  gowork за Cloudflare + reCAPTCHA; ходим через Jina/Tavily → их
  датацентровые IP получают капчу (наш дев-IP резидентный, проходит; в
  облаке будем датацентровым → тоже бан). `extract_website_text` уже
  ловит капчу guard'ом и не врёт «отзывов нет».
  Playwright-спайк (2026-07-09): headless с резидентного IP проходит
  Cloudflare+reCAPTCHA; профиль `/{slug},{id}` редиректит на
  `/opinie_czytaj,{id}`; АГРЕГАТ (балл 2/5, «N ocen», «Opinie(N)»,
  фасеты Pozytywne/Negatywne) рендерится сразу; ТЕЛА отзывов стримятся
  RSC-запросом `opinie_czytaj?_rsc=…` — это robots-Disallow
  (`/opinie_czytaj*?*`) + консент-гейт. robots РАЗРЕШАЕТ профиль/главный
  документ, запрещает именно ленту отзывов.
  Юр-рамка (принято): анализ ПОВЕРХ отзывов + агрегат без хранения
  сырья (модель ground.news) сильно ниже по риску, чем копирование
  (copyright/RODO/sui-generis); НО доступ — отдельная ось, обход
  bot-protection (решение капчи / подмена residential-IP) НЕ делаем.
  Лицензии/API у gowork нет (их B2B = управление репутацией для самих
  проверяемых компаний; контакт `wspolpraca@gowork.pl`); оператор
  судебно-агрессивен (~200 споров, в осн. выиграны); отзывы
  манипулируемы (накрутка/индустрия удаления) — взвешивать осторожно.
  **Развилка по надёжной вытяжке (решение владельца, отложено):**
  managed-browser API (Browserless/ScrapingBee/BD Web Unlocker ~$49/мес,
  residential + решает Cloudflare, БЕЗ Playwright у нас в облаке) /
  наш Playwright на residential-egress / остаёмся best-effort.
  Итерация 1 (сделано): агрегат-счётчик из веб-выдачи как слабый
  неофициальный сигнал в промпте, без прямого fetch gowork и без
  браузера. Проверено: число «N opinii» приходит стабильно, а балл
  (X/5) и тональность — нет, плюс несколько профилей = риск тёзок/
  филиалов (сверять город/адрес). Тела отзывов НЕ тянем (сложно +
  рискованно + не нужны под модель «анализ поверх»).

## Звезда качества

- **«Готовность к будущему»** — решением владельца НЕ ось звезды (слишком
  неочевидный расчёт), а отдельный текстовый блок отчёта, когда дойдём:
  PKD → отраслевые сдвиги ЕС → публичные сигналы адаптации компании.
  Урок пробы на MPSYSTEM: блок меряет только публично видимые сигналы —
  инсайдерские факты в него не попадают.
- **Поле «что вы знаете о компании»** — пользовательский ввод как вход
  отчёта: инсайдерский контекст (слухи, детали оффера), который агент
  проверяет и вплетает в анализ.
- **Калибровка весов** — веса формул осей v1 произвольные; пересмотреть
  на корпусе накопленных отчётов.
- **Владельцы в скоринге** — сейчас владельцы (CRBR) и структура
  управления (dział 2) идут только в текст отчёта. Завести в оси звезды
  детерминированные сигналы: иностранные бенефициары у молодой фирмы с
  минимальным капиталом, несовпадение владельцев и правления, просрочка
  подачи в CRBR (пусто + фирма старше дедлайна ~14 раб. дней = красный
  флаг; пусто у свежей — норма; пусто у листингованной S.A. — освобождение,
  не флаг). Отложено до калибровки весов.

## UI / дизайн

- **Тёмная тема** — светлая «документная» сейчас осознанный выбор
  дизайн-системы; вернуться при запросе пользователей.
- **Айдентика**: фирменный шрифт (вендорить woff2, CDN в рантайме
  запрещён) + логотип на основе фавикона-пентагона.
- **Граф бенефициаров (фирмы + люди)** — визуализация сети владения.
  Узлы: фирмы/люди; рёбра: владение (доля), правление/прокура, пунктир
  «общий адрес»; риск закодирован в вид узла (молодая+мин.капитал —
  красная, хаб-владелец крупнее, банкротство/расхождение — бейдж), т.е.
  это карта риска, а не просто «кто с кем». Согласовано (2026-07-10):
  СНАЧАЛА модель данных nodes/edges в evidence (дёшево, это форма того,
  что уже тянем, питает и текстовые сигналы); ХОЛСТ — потом.
  Холст = **статичный SVG на сервере**, НЕ интерактивная либа. Причины:
  (а) вписывается в документную тему и PDF-экспорт, no-CDN; (б) бизнес-
  модель — деньги за КАЖДЫЙ отчёт, значит отчёт = разовый снимок-артефакт,
  а не эксплорер: «живой» drill-down и провоцирует рекурсию по реестрам
  (=деньги на fan-out), и отдаёт неограниченную ценность с одной покупки
  (см. принцип [[report-as-paid-artifact]] в памяти). Рисовать УСЛОВНО —
  только на нетривиальной сети (у обычной фирмы граф = скучная звезда из
  3 узлов, шум). Богатая структура (кластеры/кольца/общие номиналы)
  требует depth-2 = отложенная рекурсия. PII: визуализированный
  кросс-компанийный профиль физлица — та же чувствительность, PESEL вне.

## Инфраструктура

- **История отчётов**: БД вместо файлов в `reports/` (Evidence Store),
  очередь задач для health-check'ов, PDF-экспорт отчёта.
- **Кэш отчётов по KRS** (TTL ~7 дней): повторный запрос — мгновенно и
  бесплатно. Отложено до масштаба: при `min-instances=0` файловый кэш
  стирается холодным стартом, а на объёме валидации пересечений по KRS
  мало. Вернуться вместе с `min-instances=1` (in-memory dict проще файлов).
- **Вейтлист-хранилище с доступом (PII)**: email из фейк-дора сейчас
  пишутся в выделенный лог-сток `toxic_scanner.waitlist` (нет БД при
  `min-instances=0`). До монетизации/реального сбора — перенести в
  access-controlled хранилище (БД-таблица или защищённый бакет) с
  ретеншеном и правом на удаление (GDPR: аудитория ЕС).
- **Деплой** — после него реалистичная монетизация: разовые платные
  отчёты, проверка контрагентов (не только работодателей).
